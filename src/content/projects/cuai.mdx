### 기여 역할

    - Next.js 기반의 반응형 웹 프론트엔드 개발 및 UI/UX 디자인 총괄
    - Notion API를 활용한 Headless CMS 구조 설계 및 연동 (유지보수 비용 최소화)
    - 학회원 전용 과제 제출 및 퀴즈 풀이 페이지 구현 (REST API 연동)

### 문제점 & 해결 방법

    1. **비개발자 운영진을 위한 유지보수 시스템 구축**
        - **문제**: 학회 운영진이 매년 바뀌고 웹 개발 경험이 없는 경우가 많아, 하드코딩된 텍스트나 복잡한 어드민 페이지 방식은 콘텐츠 업데이트(프로젝트, 수상 내역 등)에 지속적인 유지보수 비용이 발생할 우려가 있었습니다.
        - **해결**: Notion API를 활용하여 노션을 데이터베이스(DB)처럼 사용하는 구조를 도입했습니다. 퀴즈와 같이 로직이 필요한 부분은 별도 DB를 사용하되, 자주 바뀌는 정적 콘텐츠는 노션 페이지의 데이터를 동적으로 불러오도록 구현했습니다.
        - **결과**: 별도의 관리자 페이지 개발 공수를 줄이면서도, 운영진이 익숙한 노션 인터페이스를 통해 실시간으로 웹페이지 내용을 수정할 수 있는 지속 가능한 운영 환경을 마련했습니다.

    2. **Notion API 연동 시 보안 이슈 해결 및 데이터 구조 최적화**
        - **문제**: Notion API를 클라이언트(브라우저)에서 직접 호출할 경우, API Key가 네트워크 탭에 노출되는 보안 취약점이 있었습니다. 또한, Notion이 제공하는 Raw Data는 중첩이 깊은 Block 구조로 되어 있어, 프론트엔드 컴포넌트에서 바로 사용하기에는 로직이 불필요하게 복잡해지는 문제가 있었습니다.
        - **해결**: Next.js API Routes를 활용하여 클라이언트와 Notion API 사이에 중간 서버(Proxy Server) 역할을 하는 로직을 구현했습니다.
        - **보안 확보**: API Key 호출을 서버 사이드에서 수행하여 클라이언트에 키가 노출되지 않도록 환경 변수로 안전하게 관리했습니다.
        - **데이터 가공**: 복잡한 노션 데이터를 프론트엔드 UI에 맞는 직관적인 형태(JSON)로 가공(Normalization)하여 응답하도록 설계했습니다.
        - **결과**: 데이터베이스 관리의 편의성은 유지하면서도 보안성을 확보했고, 프론트엔드 코드는 UI 렌더링에만 집중할 수 있도록 관심사를 분리했습니다.

### 배운 점

    - 단순히 기능을 구현하는 것을 넘어, '누가 이 서비스를 관리하는가(운영진)'와 '누가 보는가(외부 인원, 학회원)'를 깊이 고민했습니다. 운영진을 위한 Notion 연동과 모바일 사용자를 위한 반응형 UI 최적화를 수행하며 실사용자(사용자, 개발자) 경험을 모두 챙기는 시각을 길렀습니다.
